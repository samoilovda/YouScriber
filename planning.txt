Текущий функционал и архитектура приложения YouScriber
======================================================
Приложение является инструментом (Knowledge Harvester) для массового скачивания и извлечения полезной текстовой информации из видео и плейлистов YouTube. 

Основной стек:
- Язык: Python
- Интерфейс: Streamlit (веб-интерфейс, работающий локально)
- Основные зависимости: yt-dlp (для парсинга и скачивания), streamlit

Ключевые функции:
1. Импорт из YouTube:
   - Извлечение списка видео из плейлистов и каналов.
   - Скачивание субтитров (vtt/srt) с приоритетом языков: английский, русский, украинский (в том числе автосгенерированных).
   - Извлечение метаданных (название, ссылка, дата загрузки, описание).
   - "YouTube Bot Bypass" — использование cookies из установленных браузеров (Chrome, Safari, Firefox и др.) для обхода капчи и блокировок YouTube.

2. Обработка локальных файлов:
   - Загрузка и обработка заранее скачанных файлов (.vtt, .srt, .json, .txt).
   - Автоматическое объединение связанных пар "субтитры + метаданные".

3. Очистка и форматирование (Core Logic):
   - Удаление WEBVTT/HTML тегов и таймкодов из субтитров.
   - Очистка от "скользящего окна" (дублирования строк из-за особенностей автосубтитров YouTube).
   - Форматирование итогового текста для скармливания нейросетям (Large Language Models), включая заголовок, ссылку, дату и очищенное описание без ссылок и хештегов.

4. Экспорт:
   - Группировка видео по плейлистам/каналам.
   - Гибкие стратегии слияния текстов: без слияния, все в один файл, чанки по ~50k символов, чанки по ~200k символов.
   - Скачивание итоговых файлов (единственный .txt файл или .zip архив).

Проблемы текущей архитектуры для десктопа:
- Интерфейс и бизнес-логика сильно перемешаны. Внутри функций обработки прямо вызываются методы Streamlit (st.error, st.session_state, progress_bar).
- Необходимость запускать локальный веб-сервер через терминал (`streamlit run`).


Варианты переноса в полноценное десктопное приложение под macOS (.app)
======================================================================

Вариант 1. Обертка (Webview / Electron wrapper) + PyInstaller
-------------------------------------------------------------
Суть: Запускать Streamlit как фоновый процесс локально, а пользователю показывать окно браузера без рамок, которое открывает localhost:8501.
- Плюсы: 
  * Минимальное вмешательство в существующий код.
  * Интерфейс остается таким же, как сейчас.
- Минусы: 
  * Приложение потребляет больше ОЗУ (т.к. запускает веб-сервер и браузерный движок).
  * Сложности с закрытием фоновых процессов при закрытии приложения.
  * Это не "настоящее" (native) десктопное приложение, оно может ощущаться неповоротливым.
- Трудозатраты: Низкие.

Вариант 2. Перенос UI на десктопный Python-фреймворк (CustomTkinter, Flet, PyQt6)
---------------------------------------------------------------------------------
Суть: Полностью отказаться от Streamlit. Написать графический интерфейс с нуля на библиотеках, предназначенных специально для десктопа, объединив их с текущей логикой `yt-dlp`. 
- Плюсы:
  * Приложение остается полностью на Python. 
  * Отличная интеграция с ОС (поддержка Dark Mode, свои окна, системные уведомления).
  * Меньшее потребление ресурсов (без локального сервера).
  * Удобная упаковка через PyInstaller в единый бинарник (.app).
- Минусы:
  * Необходим рефакторинг текущего `app.py`: жесткое разделение бизнес-логики и UI.
  * Придется верстать интерфейс заново.
- Трудозатраты: Средние.
*Рекомендация: Использовать Flet (основан на Flutter, отлично выглядит) или CustomTkinter (просто и современно).*

Вариант 3. Нативная разработка (SwiftUI + Python Backend / Swift Backend)
-------------------------------------------------------------------------
Суть: Написать визуальную часть на нативном для Apple языке Swift (используя SwiftUI). Всю скачивающую логику либо вызывать через запуск Python-процессов (Process) из Swift, либо интегрировать PythonKit, либо использовать библиотеку yt-dlp (или обертки) напрямую на Swift.
- Плюсы:
  * Идеальный пользовательский опыт (UX). Высший класс по меркам Apple/macOS.
  * Быстрая и плавная работа, нативные анимации, минимум занимаемого места.
- Минусы:
  * Смена стека технологий. 
  * Сложность настройки моста между UI на Swift и парсером на Python (или необходимость переписывать парсинг на Swift, что в случае с YouTube очень сложно поддерживать).
- Трудозатраты: Высокие.

План действий для старта (при выборе Варианта 2 или 3):
1. Изолировать слой бизнес-логики: вынести `clean_vtt_content`, `fetch_playlist_info` и `download_videos` в отдельный модуль (core.py), который ничего не знает о UI, а прогресс сообщает через callback-функции.
2. Подготовить прототип нового интерфейса на выбранной технологии (окна, кнопки, прогресс-бары).
3. Настроить скрипт сборки под macOS (PyInstaller `build.spec`), уделив особое внимание включению бинарников `yt-dlp`, если они нужны, и корректному пути к FFmpeg.
